AiBizCore Intelligence — instruction.txt
========================================

Este ficheiro explica como colocar a arquitetura a funcionar localmente e indica a função dos principais ficheiros, para facilitar manutenção ou alterações futuras.


1. Visão geral
--------------

O AiBizCore Intelligence transforma pedidos escritos em linguagem natural em blueprints técnicos para a plataforma AIBIZCore.

Fluxo geral:

1. O utilizador escreve um pedido no frontend.
2. O backend recebe o pedido através da API.
3. O router classifica a intenção do pedido, por exemplo:
   - CREATE_SYSTEM
   - CREATE_OBJECT
   - CREATE_WORKSPACE
   - UPDATE_SCHEMA
   - CHAT
4. O sistema verifica se existe uma resposta semelhante em cache.
5. Se necessário, são chamados modelos locais através do Ollama.
6. O blueprint é validado e normalizado.
7. O resultado é devolvido ao frontend.
8. Opcionalmente, o blueprint pode ser convertido em:
   - plano SQL Server;
   - plano de metadata para a framework AIBIZCore.

A arquitetura foi pensada para correr localmente, com LLMs via Ollama, evitando depender de APIs externas para a geração principal.


2. Pré-requisitos
-----------------

Antes de iniciar o projeto, confirmar que estão instalados:

- Python 3.10 ou superior
- Node.js e npm
- Ollama
- Git
- SQL Server, apenas se for necessário testar integração real com base de dados

Para confirmar:

python --version
node --version
npm --version
ollama --version


3. Instalar e preparar o Ollama
------------------------------

Iniciar o Ollama, se ainda não estiver ativo:

ollama serve

Noutro terminal, instalar os modelos usados pela arquitetura:

ollama pull qwen2.5:7b
ollama pull llama3.2:3b
ollama pull llama3.1:8b
ollama pull nomic-embed-text

Modelos usados normalmente:

- qwen2.5:7b — planner/classificação de tarefas mais estruturadas
- llama3.2:3b — geração mais leve e rápida
- llama3.1:8b — geração alternativa mais robusta
- nomic-embed-text — embeddings para cache semântica

Para testar se o Ollama responde:

ollama run llama3.2:3b

Depois de confirmar, sair com:

/bye


4. Preparar o backend
---------------------

Abrir um terminal na pasta principal do projeto:

cd AiBizCore_v4

Criar ambiente virtual:

Windows:

python -m venv .venv
.venv\Scripts\activate

macOS/Linux:

python3 -m venv .venv
source .venv/bin/activate

Instalar dependências:

pip install -r requirements.txt

Se o projeto estiver a usar uv, também pode ser usado:

uv pip install -r requirements.txt

Criar ou confirmar o ficheiro .env na raiz do backend/projeto. Exemplo sem credenciais reais:

OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3.2:3b
OLLAMA_PLANNER_MODEL=qwen2.5:7b
OLLAMA_GENERATOR_MODEL=llama3.2:3b
EMBEDDING_MODEL=nomic-embed-text

# Execução SQL real — manter desativado por defeito
ENABLE_SQL_EXECUTION=false

# Execução de metadata real na framework — manter desativado por defeito
ENABLE_FRAMEWORK_EXECUTION=false

# Configuração SQL Server — preencher apenas em ambiente seguro
DB_SERVER=<servidor>
DB_DATABASE=<base_de_dados>
DB_USERNAME=<utilizador>
DB_PASSWORD=<password>
DB_DRIVER=ODBC Driver 17 for SQL Server

Nota: não colocar passwords, connection strings reais ou ficheiros .env em commits, relatório ou vídeo.


5. Iniciar o backend
--------------------

Na pasta onde está o ficheiro api.py, executar:

uvicorn api:app --reload --host 127.0.0.1 --port 8000

Se o ficheiro principal tiver outro nome, adaptar o comando:

uvicorn main:app --reload --host 127.0.0.1 --port 8000

O backend deverá ficar disponível em:

http://127.0.0.1:8000

Endpoints principais:

- POST /api/prompt
- GET  /api/get_schema
- POST /api/save_cache
- POST /api/regression_tests
- POST /api/db-preview
- POST /api/db-plan
- POST /api/db-test-connection
- GET  /api/db-config-status
- POST /api/db-execute
- POST /api/framework-plan
- POST /api/framework-preflight
- POST /api/framework-execute


6. Preparar o frontend
----------------------

Abrir outro terminal e entrar na pasta do frontend. Dependendo da estrutura do projeto, será uma destas opções:

cd frontend

ou:

cd client

Instalar dependências:

npm install

Iniciar o frontend:

npm run dev

Abrir no browser o URL indicado pelo Vite/React, normalmente:

http://localhost:5173

Se o frontend usar outro porto, seguir o endereço mostrado no terminal.


7. Teste rápido da aplicação
----------------------------

Com backend, frontend e Ollama ativos, testar no frontend um pedido simples:

Cria apenas um objeto chamado Teste Video Produto. Campos: codigo string, nome string, preco number, dataCriacao date.

Resultado esperado:

- O router deve classificar como CREATE_OBJECT.
- O backend deve gerar um blueprint com um único objeto.
- O frontend deve apresentar o objeto com os campos indicados.

Depois testar um sistema completo:

Cria um sistema de gestão de encomendas para uma loja online. O sistema deve ter clientes, produtos, encomendas, pagamentos e entregas. Cada cliente pode fazer várias encomendas, cada encomenda pode ter vários produtos, cada encomenda tem um pagamento associado e pode ter uma entrega. Cria também workspaces e ações principais para gerir o sistema.

Resultado esperado:

- O router deve classificar como CREATE_SYSTEM.
- O blueprint deve incluir objetos, campos, relações, workspaces e ações.


8. Limpar cache e histórico
---------------------------

Se a aplicação estiver a devolver respostas antigas ou comportamentos inesperados, limpar a cache local.

Na pasta principal do projeto:

rm -f database/cache.json
rm -f database/prompt_history.json
mkdir -p database
echo '{}' > database/cache.json
echo '[]' > database/prompt_history.json

Em Windows PowerShell:

Remove-Item database/cache.json -ErrorAction SilentlyContinue
Remove-Item database/prompt_history.json -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Force database
'{}' | Set-Content database/cache.json
'[]' | Set-Content database/prompt_history.json

Depois reiniciar o backend e, se necessário, limpar também o estado no frontend através do botão de limpar/reset.


9. Integração com SQL Server
----------------------------

A integração SQL Server deve ser testada primeiro em modo seguro.

Passos normais no frontend:

1. Criar ou carregar um blueprint.
2. Clicar em Preview BD.
3. Confirmar a tabela, chave primária, campos do utilizador e campos de sistema.
4. Clicar em Preparar Plano.
5. Ver o plano em modo dry-run.
6. Só usar Executar no SQL Server num ambiente controlado.

A execução real deve ficar bloqueada por defeito. Para permitir execução, é necessário:

- configurar corretamente a ligação à BD;
- ativar ENABLE_SQL_EXECUTION=true no servidor;
- enviar a frase de confirmação exigida pela API;
- confirmar que o plano foi validado.

A frase de confirmação usada na aplicação é normalmente:

EXECUTE_SQL_SERVER

Em caso de erro durante a execução real, o sistema deve fazer rollback.


10. Integração com a framework AIBIZCore
----------------------------------------

A integração com a framework cria metadata lógica para a plataforma. Esta parte é diferente da criação física da tabela SQL.

Passos normais no frontend:

1. Criar ou carregar um blueprint.
2. Gerar o plano da framework.
3. Executar o preflight.
4. Confirmar que não existem conflitos.
5. Só executar a metadata em ambiente seguro.
6. Validar no AIBIZCore em Definições -> Objeto.

A metadata pode incluir:

- CSYSObject — objeto lógico
- CSYSObjectField — campos do objeto
- CSYSView — vista/listagem
- CSYSAction — ações associadas
- CSYSObjectAction — ligação entre objeto e ações
- CSYSPermission — permissões
- Workspaces e ligações associadas, quando aplicável

A execução real deve ficar bloqueada por defeito. Para permitir execução, é necessário:

- ativar ENABLE_FRAMEWORK_EXECUTION=true;
- usar a frase de confirmação correta;
- passar no preflight;
- garantir que não há conflitos com objetos, tabelas ou metadata já existentes.

A frase de confirmação usada na aplicação é normalmente:

EXECUTE_FRAMEWORK_METADATA


11. Principais ficheiros e função de cada um
--------------------------------------------

Nota: os nomes podem variar ligeiramente conforme a versão do projeto. Esta secção descreve a organização lógica da arquitetura.

api.py
- Define a API FastAPI.
- Expõe os endpoints usados pelo frontend.
- Recebe prompts, schemas e pedidos de integração SQL/framework.
- É o ponto de entrada do backend.

router.py
- Classifica a intenção do pedido do utilizador.
- Decide se o pedido deve seguir para CREATE_SYSTEM, CREATE_OBJECT, CREATE_WORKSPACE, UPDATE_SCHEMA ou CHAT.
- Deve ser alterado quando for necessário melhorar regras de routing ou adicionar novas intenções.

main.py
- Pode funcionar como orquestrador principal, dependendo da versão.
- Chama o router e encaminha o pedido para o handler correto.
- Mantém ou recebe o schema ativo enviado pelo frontend.

create_system_handler.py
- Trata pedidos de criação de sistemas completos.
- Coordena geração de objetos, relações, workspaces e ações.
- Usar quando o prompt descreve um domínio completo.

create_object_handler.py
- Trata pedidos de criação de um único objeto.
- Deve gerar apenas um objeto com os seus campos.
- É útil para integração simples com SQL Server e framework.

update_schema_handler.py
- Trata pedidos incrementais sobre um blueprint já existente.
- Exemplo: adicionar campo, renomear objeto, criar relação, remover elemento.
- Se estiver instável, verificar extração de JSON, regras determinísticas e fallback do LLM.

planner.py
- Constrói o plano intermédio a partir do pedido do utilizador.
- Normalmente usa um modelo mais forte/estruturado, como qwen2.5:7b.
- Alterar quando for necessário mudar a forma como a arquitetura decompõe pedidos complexos.

object_generator.py
- Gera objetos e campos.
- Alterar para melhorar nomes de objetos, tipos de campos ou estrutura base.

relation_generator.py
- Gera relações entre objetos.
- Alterar para melhorar cardinalidades, nomes de relações ou validação de entidades existentes.

workspace_generator.py
- Gera workspaces/perfis/áreas de utilização.
- Alterar quando se quiser mudar a lógica dos perfis gerados.

action_generator.py
- Gera ações de negócio e ações básicas.
- Exemplo: criar, listar, atualizar, arquivar, aprovar, cancelar.
- Alterar quando se quiser controlar melhor as ações criadas por domínio.

canonical_schema.py
- Define ou normaliza o formato canónico do blueprint.
- Garante que os dados seguem uma estrutura esperada.
- Alterar com cuidado, porque afeta frontend, validação, SQL e framework.

models.py / schema_models.py
- Define modelos Pydantic usados para validar a estrutura dos dados.
- Alterar quando forem adicionados novos campos obrigatórios ou novas entidades internas.

validator.py
- Valida o blueprint depois da geração.
- Remove, corrige ou rejeita estruturas inválidas.
- Alterar quando for necessário endurecer ou suavizar regras de validação.

semantic_rules.py
- Contém regras determinísticas e semânticas.
- Útil para normalizar tipos, nomes, relações e ações.
- Alterar quando um erro for previsível e não deve depender do LLM.

evaluator.py
- Avalia a qualidade do blueprint.
- Pode calcular score e identificar problemas de completude ou consistência.
- Alterar quando se quiser ajustar critérios de qualidade.

cache.py / semantic_cache.py
- Gere cache semântica com embeddings.
- Evita repetir geração para pedidos semelhantes.
- Usa nomic-embed-text ou modelo equivalente.
- Alterar se for necessário mudar threshold, estratégia de reutilização ou armazenamento.

storage.py
- Guarda histórico, schemas, cache ou resultados intermédios.
- Alterar se a estrutura de persistência local mudar.

database/cache.json
- Guarda respostas em cache.
- Pode ser apagado em caso de respostas antigas ou inconsistentes.

database/prompt_history.json
- Guarda histórico de prompts.
- Pode ser apagado durante testes ou antes de uma demonstração.

sql_adapter.py / db_adapter.py
- Converte o blueprint em estrutura SQL Server.
- Define tabelas, colunas, tipos, chaves e campos de sistema.
- Alterar quando for necessário mudar o formato das tabelas físicas.

db_preview.py
- Gera uma pré-visualização da estrutura SQL.
- Não deve executar alterações reais.
- Alterar quando for necessário mudar o que aparece no Preview BD.

db_plan.py
- Gera plano SQL em modo dry-run.
- Mostra o que seria executado sem alterar a base de dados.
- Alterar para modificar a forma como as instruções SQL são planeadas.

db_executor.py
- Executa alterações reais em SQL Server.
- Deve manter proteções fortes: confirmação, variável de ambiente e rollback.
- Alterar com muito cuidado.

framework_object_planner.py
- Gera o plano de metadata para a framework AIBIZCore.
- Mapeia objetos do blueprint para CSYSObject, CSYSObjectField, CSYSView, CSYSAction, permissões e estruturas relacionadas.
- Alterar quando a metadata gerada precisar de seguir outro padrão da framework.

framework_metadata_preflight.py
- Valida o plano antes de executar metadata real.
- Verifica conflitos, tabelas, objetos existentes e condições necessárias.
- Alterar quando forem necessárias novas validações de segurança.

framework_metadata_executor.py
- Executa a criação ou atualização de metadata na framework.
- Deve respeitar confirmação explícita e variável de ambiente.
- Alterar com muito cuidado, porque escreve na metadata real.

regression_tests.py
- Executa testes de regressão com prompts esperados.
- Útil para confirmar se alterações no router, prompts ou validação não partiram funcionalidades antigas.
- Atualizar sempre que forem adicionados novos cenários importantes.

frontend/src/App.jsx ou frontend/src/App.tsx
- Componente principal do frontend.
- Liga chat, estado global, canvas e chamadas à API.

frontend/src/components/*
- Componentes visuais do frontend.
- Inclui chat, canvas, cartões, botões, painéis de BD/framework, etc.

frontend/src/services/api.js ou api.ts
- Centraliza chamadas do frontend para o backend.
- Alterar quando forem adicionados ou renomeados endpoints.

frontend/src/components/Canvas* ou Blueprint*
- Renderiza o blueprint visual.
- Alterar quando se quiser mudar a forma como objetos, relações, workspaces ou ações aparecem.

.env
- Guarda configuração local e variáveis sensíveis.
- Não deve ser partilhado publicamente.

requirements.txt
- Lista dependências Python do backend.
- Atualizar quando forem adicionadas novas bibliotecas.

package.json
- Lista dependências e scripts do frontend.
- Atualizar quando forem adicionadas novas bibliotecas JS/React.


12. Como modificar funcionalidades comuns
-----------------------------------------

Alterar o modelo LLM usado:

- Editar .env.
- Mudar OLLAMA_MODEL, OLLAMA_PLANNER_MODEL ou OLLAMA_GENERATOR_MODEL.
- Reiniciar o backend.

Alterar a classificação dos pedidos:

- Editar router.py.
- Adicionar ou ajustar regras para CREATE_SYSTEM, CREATE_OBJECT, UPDATE_SCHEMA, etc.
- Testar com prompts curtos e longos.

Melhorar criação de objetos:

- Editar create_object_handler.py e object_generator.py.
- Confirmar que o output segue o canonical_schema.
- Testar com prompts de um único objeto.

Melhorar criação de sistemas completos:

- Editar create_system_handler.py, planner.py e generators.
- Verificar se objetos, relações, workspaces e ações continuam consistentes.

Melhorar edição incremental:

- Editar update_schema_handler.py.
- Adicionar regras determinísticas para operações previsíveis, por exemplo:
  - adicionar campo;
  - renomear objeto;
  - adicionar relação;
  - remover campo.
- Evitar depender apenas do LLM para JSON diff.

Alterar validação:

- Editar validator.py, semantic_rules.py e schema_models.py.
- Correr regression_tests depois da alteração.

Alterar preview SQL:

- Editar db_preview.py e sql_adapter.py.
- Confirmar que não existe execução real nesta fase.

Alterar execução SQL:

- Editar db_executor.py.
- Manter obrigatórios: confirmação, variável de ambiente, validação e rollback.

Alterar metadata da framework:

- Editar framework_object_planner.py, framework_metadata_preflight.py e framework_metadata_executor.py.
- Testar primeiro em dry-run/preflight.
- Confirmar depois em AIBIZCore -> Definições -> Objeto.

Alterar frontend:

- Editar componentes em frontend/src/components.
- Editar chamadas API em frontend/src/services.
- Confirmar que o backend continua a receber current_schema corretamente.


13. Problemas frequentes
------------------------

Problema: o backend não arranca.
Soluções:
- Confirmar ambiente virtual ativo.
- Confirmar dependências instaladas.
- Confirmar se o ficheiro correto é api.py ou main.py.
- Verificar se o porto 8000 já está ocupado.

Problema: o frontend não comunica com o backend.
Soluções:
- Confirmar que o backend está em http://127.0.0.1:8000.
- Confirmar variável de API no frontend, se existir.
- Verificar erros CORS no browser.

Problema: o LLM não responde.
Soluções:
- Confirmar que ollama serve está ativo.
- Confirmar que o modelo foi instalado com ollama pull.
- Testar manualmente com ollama run llama3.2:3b.

Problema: a aplicação devolve respostas antigas.
Soluções:
- Limpar database/cache.json.
- Limpar database/prompt_history.json.
- Reiniciar backend.
- Limpar estado do frontend.

Problema: UPDATE_SCHEMA falha a extrair JSON.
Causa provável:
- O LLM respondeu com JSON inválido, comentários ou reticências.
Soluções:
- Usar prompts incrementais mais simples.
- Adicionar regra determinística em update_schema_handler.py.
- Melhorar parser/extrator JSON.
- Não depender apenas do LLM para operações previsíveis.

Problema: execução SQL está bloqueada.
Causa provável:
- Proteção de segurança ativa.
Soluções:
- Confirmar ENABLE_SQL_EXECUTION.
- Confirmar frase de confirmação.
- Confirmar ligação à base de dados.
- Confirmar plano validado.

Problema: metadata da framework não executa.
Soluções:
- Confirmar ENABLE_FRAMEWORK_EXECUTION.
- Executar preflight.
- Confirmar que não existem conflitos.
- Confirmar permissões e ligação correta à base de dados/framework.


14. Prompts recomendados para teste
-----------------------------------

Criar objeto simples:

Cria apenas um objeto chamado Teste Video Produto. Campos: codigo string, nome string, preco number, dataCriacao date.

Criar sistema completo:

Cria um sistema de gestão de encomendas para uma loja online. O sistema deve ter clientes, produtos, encomendas, pagamentos e entregas. Cada cliente pode fazer várias encomendas, cada encomenda pode ter vários produtos, cada encomenda tem um pagamento associado e pode ter uma entrega. Cria também workspaces e ações principais para gerir o sistema.

Prompt incremental simples:

Renomeia o objeto Produto para Artigo.

Prompt incremental alternativo:

adiciona campo estado em Encomenda

16. Notas de segurança
----------------------

- Nunca partilhar o ficheiro .env.
- Nunca mostrar passwords ou connection strings em vídeo.
- Não ativar execução real em ambiente de demonstração sem necessidade.
- Preferir preview e dry-run para testes.
- Executar SQL/framework real apenas em ambiente controlado.
- Confirmar sempre o objeto criado na framework através de Definições -> Objeto.
